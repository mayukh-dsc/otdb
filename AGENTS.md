# AGENTS.md — Guidance for AI assistants and contributors

This document helps humans and automated agents work safely and effectively on **OTDB (Old Temples Database)**.

## What this project is

- **Purpose:** Interactive exploration of historically significant temples (India and Southeast Asia) on a map, with detail pages, a terminology handbook, and tag-based discovery.
- **Stack:** Next.js 16 (App Router), React 19, TypeScript, Tailwind CSS v4, Leaflet (`react-leaflet` + clustering).
- **Data:** ~128 curated temples. Primary store is **`data/temples.db`** (dev/pipeline only); **`data/temples.json`** is the runtime data source (regenerated by the pipeline).
- **Deployment:** Vercel (free tier). Images served locally from `public/images/temples/`.

## Run locally

```bash
npm install
npm run download-images   # fetch temple images to public/images/temples/
npm run dev
```

App: [http://localhost:3000](http://localhost:3000).

- If **`data/temples.db`** is missing or corrupt, recreate it:

  ```bash
  npx tsx scripts/pipeline/migrate-to-sqlite.ts
  ```

- If images are missing, re-run `npm run download-images`.

## Architecture

```
Pipeline (offline)              Build (Vercel)              Runtime
─────────────────               ──────────────              ───────
temples.db ──export──> temples.json ──────────> API routes (JSON read)
temples.db ──download──> public/images/ ──────> Edge CDN (static files)
```

- **SQLite** (`better-sqlite3`) is a **dev dependency only** — used by the pipeline scripts for enrichment, transforms, and data curation.
- **At runtime**, the app reads `data/temples.json` (in-memory) and serves local images from `public/images/temples/`.
- **On Vercel**, the build command downloads images then runs `next build`. Images are served via the global edge CDN.

## API (server)

- `GET /api/temples` — list temples; query params: `religion`, `country`, `style`, `hasVisualization=true`, `tag`
- `GET /api/temples/[id]` — single temple (full JSON)
- `GET /api/temples/by-tag` — `?tag=...` for filtered list, or no `tag` for tag counts

API routes read from `data/temples.json` via `lib/db.ts`. No database connection at runtime.

## Data pipeline

Entry point:

```bash
npx tsx scripts/pipeline/run.ts
npx tsx scripts/pipeline/run.ts --export-json   # also refresh data/temples.json
npx tsx scripts/pipeline/run.ts --dry-run
npx tsx scripts/pipeline/run.ts --source wikidata
npx tsx scripts/pipeline/run.ts --temple q916943
```

- **Schema:** `scripts/pipeline/schema.sql`
- **DB helpers:** `scripts/pipeline/db.ts` (used by migration and sources; differs from `lib/db.ts` which is the app data layer)
- **Image download:** `npm run download-images` or `npx tsx scripts/pipeline/download-images.ts [--force]`

## Images

- Temple images are resolved through `lib/templeImage.ts` helper to keep storage provider swappable.
- Candidate order is: Blob/CDN (`NEXT_PUBLIC_TEMPLE_IMAGE_BASE_URL`) -> local `/images/temples/<id>.jpg` -> remote Wikimedia fallback.
- Temple images can be uploaded to Vercel Blob with:
  - `npm run upload-images:blob`
  - then set `NEXT_PUBLIC_TEMPLE_IMAGE_BASE_URL` to the printed base URL.
- Local download remains available:
  - `npm run download-images` to populate `public/images/temples/`.
- The `public/images/temples/` directory is **gitignored** — images are reproducible via `npm run download-images`.
- On Vercel, builds no longer fetch Wikimedia images. `vercel.json` runs `next build` only.
- Floor plans and blueprints (from `visualization.floorPlanUrls`, etc.) still use remote URLs since they are rare and not bulk-downloaded.

## Identifiers and types

- Temple **`id`** is typically a lowercase Wikidata-style id (e.g. `q916943`).
- Shared types live in **`lib/types.ts`**. Keep API responses aligned with `Temple` and nested optional fields (`engineering`, `archaeology`, `visualization`, `graphTags`).

## Similarity connections

- `lib/similarityEdges.ts` computes undirected edges between temples sharing attributes (style, complex, dynasty, religion, century).
- The map renders these as dashed polylines via `react-leaflet` `Polyline` in a custom pane (z-index 350, under markers).
- Toggle in the home page header controls the active similarity mode.

## Testing

```bash
npm run test:e2e          # Playwright tests (starts dev server automatically)
npx playwright install    # first-time browser install
```

Tests in `e2e/` validate image loading, placeholder fallbacks, and page rendering.

## What to avoid

- Do not commit **secrets** (API keys, `.env` with real credentials). Use `.env.local` (gitignored) for local keys if you add paid APIs later.
- Avoid **destructive** pipeline runs on production data without a backup; `data/temples.db` is the pipeline store. Prefer git history on committed `temples.json` for rollback.
- Do not import **`better-sqlite3`** from any file under `app/` or `lib/` — it is a dev dependency and will fail on Vercel.
- Do not couple **new subsystems** (e.g. audiobooks) directly into the temple DB without an explicit schema decision; prefer separate manifests or tables when lifecycles differ.

## Code quality

- **Lint:** `npm run lint`
- **Build:** `npm run build` before merging larger changes
- **Test:** `npm run test:e2e` for end-to-end validation

## Deployment

- **Platform:** Vercel (Hobby / free tier)
- **Build command:** `npm run download-images && next build` (configured in `vercel.json`)
- **Install command:** `npm install --include=dev` (needed for `tsx` used by download script)
- Connect via GitHub for automatic deploys on push to `main`.

## UX / product notes

- **Map** is the primary surface; **temple detail** is at `/temple/[id]`.
- **Handbook** (`/handbook`) uses `data/handbook.json`; terms link to `/temples?tag=...` where tags align with `graphTags` on temples.
- **Similarity lines** connect temples on the map by shared attributes (dynasty, style, complex, religion, century).

When making large UI changes, keep **mobile** and **map + panel** layout in mind; test a few temple ids with and without images.
