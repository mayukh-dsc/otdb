# AGENTS.md — Guidance for AI assistants and contributors

This document helps humans and automated agents work safely and effectively on **OTDB (Old Temples Database)**.

## What this project is

- **Purpose:** Interactive exploration of historically significant temples (India and Southeast Asia) on a map, with detail pages, a terminology handbook, and tag-based discovery.
- **Stack:** Next.js 16 (App Router), React 19, TypeScript, Tailwind CSS v4, Leaflet (`react-leaflet` + clustering), **SQLite** (`better-sqlite3`) for temple data.
- **Data:** ~128 curated temples. Primary store is **`data/temples.db`**; **`data/temples.json`** is an export for distribution and readable diffs (regenerated by the pipeline).

## Run locally

```bash
npm install
npm run dev
```

App: [http://localhost:3000](http://localhost:3000).

- If **`data/temples.db`** is missing or corrupt, recreate it:

  ```bash
  npx tsx scripts/pipeline/migrate-to-sqlite.ts
  ```

## Repository layout (high level)

| Path | Role |
|------|------|
| `app/` | Routes: `/` (map), `/handbook`, `/temples`, `/temple/[id]`, API under `app/api/temples/` |
| `components/` | UI: `MapView`, `TempleDetailPanel`, `FeatureBadge`, `components/temple/*` |
| `lib/types.ts` | TypeScript types: `Temple`, `TempleEngineering`, `TempleArchaeology`, `TempleVisualization`, etc. |
| `lib/db.ts` | Read-only SQLite access for API routes (opens `data/temples.db`) |
| `data/temples.db` | Source of truth for the running app |
| `data/temples.json` | JSON export (pipeline output) |
| `data/handbook.json` | Curated glossary terms for `/handbook` |
| `data/pipeline-state.json` | Pipeline metadata and semver-style `dataVersion` |
| `scripts/pipeline/` | Orchestrator, schema, migration, sources, transforms |

## API (server)

- `GET /api/temples` — list temples; query params: `religion`, `country`, `style`, `hasVisualization=true`, `tag`
- `GET /api/temples/[id]` — single temple (full JSON)
- `GET /api/temples/by-tag` — `?tag=...` for filtered list, or no `tag` for tag counts

API routes use **`lib/db.ts`** and must not expose internal-only fields if you add any (e.g. raw `data_sources` if reintroduced for clients).

## Data pipeline

Entry point:

```bash
npx tsx scripts/pipeline/run.ts
npx tsx scripts/pipeline/run.ts --export-json   # also refresh data/temples.json
npx tsx scripts/pipeline/run.ts --dry-run
npx tsx scripts/pipeline/run.ts --source wikidata
npx tsx scripts/pipeline/run.ts --temple q916943
```

- **Schema:** `scripts/pipeline/schema.sql`
- **DB helpers:** `scripts/pipeline/db.ts` (used by migration and sources; differs from `lib/db.ts` which is app read-only)

Older one-off scripts may still exist under `scripts/` (e.g. `seed-from-wikidata.ts`); prefer the **`scripts/pipeline/`** flow for new work unless maintaining legacy scripts.

## Identifiers and types

- Temple **`id`** is typically a lowercase Wikidata-style id (e.g. `q916943`).
- Shared types live in **`lib/types.ts`**. Keep API responses aligned with `Temple` and nested optional fields (`engineering`, `archaeology`, `visualization`, `graphTags`).

## Images and external assets

- Temple **`imageUrl`** values often point at **Wikimedia Commons** / Wikipedia uploads. Hotlinking can fail (403, redirects). The UI may use `<img referrerPolicy="no-referrer" />` for some cases.
- **`next.config.ts`** defines `images.remotePatterns` for Commons hosts. If you switch to `next/image` for new UI, add any new hostnames there.
- When debugging broken images: verify the URL in the browser, check Network tab status, and confirm the URL still exists in `temples` rows or JSON export.

## What to avoid

- Do not commit **secrets** (API keys, `.env` with real credentials). Use `.env.local` (gitignored) for local keys if you add paid APIs later.
- Avoid **destructive** pipeline runs on production data without a backup; `data/temples.db` is the live store. Prefer git history on committed `temples.json` + DB backups for rollback (see project docs / future `docs/`).
- Do not couple **new subsystems** (e.g. audiobooks) directly into the temple DB without an explicit schema decision; prefer separate manifests or tables when lifecycles differ.

## Code quality

- **Lint:** `npm run lint`
- **Build:** `npm run build` before merging larger changes

## UX / product notes

- **Map** is the primary surface; **temple detail** is at `/temple/[id]`.
- **Handbook** (`/handbook`) uses `data/handbook.json`; terms link to `/temples?tag=...` where tags align with `graphTags` on temples.

When making large UI changes, keep **mobile** and **map + panel** layout in mind; test a few temple ids with and without images.
